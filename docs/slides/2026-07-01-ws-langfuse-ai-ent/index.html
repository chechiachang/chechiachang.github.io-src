核心問題

多數團隊已經有：

  • trace / logging
  • prompt 管理
  • agent framework（LangChain / AutoGen / others）
  • observability 工具（Langfuse）

但仍無法回答：

這次 model 或 framework 升級，該不該上 production？

關鍵誤解

Observability != Decision System

Observability 只能回答：

發生了什麼？

Decision System 需要回答：

我應不應該改？
這次改動是不是變好？

LAB1：啟動 LLM O11y Stack

目標：把本地端觀測平台跑起來，先確保 trace 能收集。

cd llm-o11y
cp .env.example .env
docker compose up -d

LAB1：確認服務健康

請確認：

  • Langfuse UI 可開啟
  • Gateway / API 可回應
  • demo app 能發送一筆 trace
docker compose ps

LAB1：做第一筆 trace

make demo-trace

觀察重點：

  • prompt / response
  • token usage
  • latency

Evaluation：把主觀變可比較

建議組合：

  • LLM-as-a-judge（快速、可擴）
  • rule-based checks（格式、schema、constraints）
  • task rubric（符合業務定義的好壞）

LAB2：加入 LLM-as-a-judge

目標：針對既有 traces 自動評分。

make eval-run

LAB2：加上 Rule-based Checks

常見檢查：

  • JSON schema 是否合法
  • 必要欄位是否存在
  • 關鍵關鍵字是否命中
make eval-rules

Dataset：從真實流量累積

不是 benchmark 優先，而是 production-like data：

  • 真實 query
  • 真實 failure cases
  • 真實 edge cases

流程：

trace -> triage -> label -> dataset

LAB3：從 traces 產生 dataset

make dataset-export

輸出目標：

  • baseline dataset
  • failure-focused dataset

LAB3：資料集分層

最低限度建議分三層：

  • smoke（快速）
  • core（關鍵流程）
  • edge（高風險案例）

Regression：比較不能靠感覺

固定同一份 dataset 比較不同：

  • model
  • prompt strategy
  • agent framework

輸出指標：

  • quality delta
  • latency delta
  • cost delta

LAB4：跑 regression

make regression-run

LAB4：設定 Decision Gate

只有同時滿足才允許 deploy：

  • quality >= baseline
  • no critical regression
  • latency within SLO
  • cost within budget
make decision-gate

Failure Taxonomy

先分類，再修復：

  • hallucination
  • tool misuse
  • reasoning error
  • format/schema break

最小可行架構

Agent Runtime
  -> Tracing (Langfuse)
  -> Evaluation
  -> Dataset Store
  -> Regression Runner
  -> Decision Policy

核心訊息

沒有 evaluation：

LLM upgrade = gambling

有 feedback loop：

Observe -> Evaluate -> Regress -> Decide
